# Pi Backup Manager — Session Notes 2026-03-19

## Bugs Fixed

### 1. iSCSI Block Device Detection (Login)
**Problem:** After iSCSI login, the block device field auto-populated incorrectly (e.g. `/dev/sda11` instead of `/dev/sda1`) because the code took `ls -1t /dev/sd*` output and blindly appended `"1"`.

**Fix:** Replaced `ls -1t /dev/sd*` approach entirely. Now uses `iscsiadm -m session -P 3` to map IQN → block device reliably. The login endpoint retries up to 5 seconds for the device to appear in sysfs.

### 2. iSCSI Auto-Population of Fields
**Problem:** If an iSCSI session was already connected, the IQN and block device fields were blank — user had to know and type them manually.

**Fix:** `/api/iscsi/sessions` now returns `device` field for each session. When the iSCSI destination panel opens, if an active session exists, IQN and block device fields are auto-populated and steps 2 & 3 are unlocked.

### 3. runtipi-reverse-proxy Left in "Created" State After Backup
**Problem:** When `runtipi-cli start` runs after backup, Docker Compose starts containers in dependency order. If `runtipi` is slow to pass its health check, Compose aborts early and leaves `runtipi-reverse-proxy` in "Created" (never started). This causes all apps to show offline in Runtipi.

**Fix:** Added explicit check in the generated backup script's restart block — after runtipi passes health check, the script checks if `runtipi-reverse-proxy` is running and starts it if not.
> **Note:** Regenerate and re-save the script in the Generate tab to pick up this fix.

**Immich stack** (4 containers): `network 017dedf9454bac6f1c2cb327251a818b70b5b1baa0c590522ce49c0ea7b4d6b1 not found`
- Root cause: Runtipi's generated compose file references the Docker network by **ID** (not name). After `runtipi-cli start`, the network is recreated with a new ID. Both `docker start` and `docker compose up -d` fail because the network is defined as `external: true`.

- Root cause: Same — external network gone after Runtipi restart. `qbittorrent-1` fails as a consequence (depends on gluetun network namespace).

**Immediate fix:** Started both stacks manually via Runtipi dashboard — all containers now running.

### Bug Fixed (Post-Backup)

### 4. Immich/qbitgluetun Fail to Restart After Backup
**Problem:** `smart_start`'s `docker compose up -d` fallback also fails for Runtipi-managed app containers because their compose files reference Docker networks as `external: true`. Only Runtipi itself can recreate those networks.

**Fix:** Three-tier fallback in `smart_start`:
1. `docker start` (as before)
2. `docker compose up -d` (as before, now uses `PIPESTATUS[0]` to correctly detect failure through the `| sed` pipe)
3. **New:** If compose fails AND container is in `/data/apps/`, use Runtipi REST API:
   - `POST /api/auth/login` → session cookie saved to `/tmp/_pbm_tipi_cookies`
   - `GET /api/apps/installed` → find URN matching `appName` extracted from compose path
   - `POST /api/app-lifecycle/{urn}/start`
   - Login called once in `restart_containers` before the start loop

**New config fields** (Config tab, under Runtipi Directory):
- Runtipi URL (default `http://localhost`)
- Runtipi Username
- Runtipi Password *(stored in generated script — file permissions matter)*

> **Action required:** Enter Runtipi credentials in Config tab, save, then regenerate and re-save script in Generate tab.

### Bug Fixed (Post-Session)

### 5. "No Backup Yet" After Service Restart
**Problem:** `_backup_status` is in-memory only — resets to `{result: None}` on every service restart, so the dashboard badge always showed "No backup yet" even after a successful backup.

**Fix:** Added `_apply_sentinel_fallback(cfg, out)` helper in `api_backup_last`. If the log file yields no result (empty log, no marker, or no success/fail line), the function checks for `/mnt/backups/.image_initialised`. If the sentinel exists, it sets `result: "success"` and uses the image file's mtime as the timestamp. Called at all three early-return points in `api_backup_last`.
